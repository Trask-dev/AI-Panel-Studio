# 开发过程思路 & 工作流说明

## 一、开发流程

本项目使用 **Claude Code CLI** 作为交互终端，后端调用 **Deepseek V4 Pro** 模型执行代码生成、测试、调试。前端使用 React + Vite，后端使用 Python FastAPI + SQLite。

### 1.1 工作流闭环

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  SDD     │ →  │  DDD     │ →  │  TDD     │ →  │  E2E     │
│ 契约设计  │    │ 组件构建  │    │ 测试驱动  │    │ 质量闭环  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
     ↑                                                 │
     └─────────────── 回归审计 ←───────────────────────┘
```

每个阶段的决策逻辑：

**SDD（规格驱动）**：不写代码，先输出 `domain-model.md` → `database_schema.sql` → `openapi.yaml`。AI 按"需求文档 → 领域实体 → 数据库 → API 契约"逐层推演。关键约束：SQL 必须用 `sqlite3` 验证语法，OpenAPI 必须用 Pydantic Schema 强类型。

**DDD（设计驱动）**：不写 JS，先输出 ASCII 布局草图 + 组件树 + CSS Token 体系。分 11 步增量搭建——每一步只做一件事，验收通过才进入下一步。关键约束：禁止整页滚动，各区域独立 `overflow-y: auto`。

**TDD（测试驱动）**：先写失败测试 → 最小代码通过 → 重构。采用"批量 TDD"策略——一次性写出 7-12 个测试函数，再一次性写出完整业务实现。关键约束：测试不通过不写业务代码，每轮回归 70+ 用例。

**E2E（端到端）**：全链路排查（后端 SSE → 前端 EventSource → Store → UI 渲染），用 F12 Network 面板验证数据流。发现系统性缺陷（如共识引擎缺失）时，回溯到 SDD 重新建立契约。

### 1.2 多范式切换

项目并非线性推进——当 E2E 阶段发现"共识/分歧完全不显示"时，立即启动 `/superpowers/systematic-debugging` 追溯根因：`asyncio.Queue` 跨线程故障 → 重写 SSEManager 为线程安全版。当发现"发言顺序机械轮流"与 SDD 矛盾时，回溯到 `orchestrator.py` 重构 `run_round()` 为 4 阶段分布式 Agent 模型。

## 二、与 AI 协同中的典型问题及解决路径

### 问题 1：AI 的"修补惯性"——不断修症状而非追溯根因

**表现**：共识/分歧面板多次"修复"后仍不显示。每次都是"加一个 fetch""改一个 CSS 的 display"，但问题反复出现。

**根因**：三层断裂同时存在——(1) 后端 `asyncio.Queue` 跨线程 push 丢失，(2) `analyze_and_persist` 无条件 SET is_active=0 导致旧数据被清空，(3) 前端 `initDiscussion` 重置 consensus 为空数组但 AnalysisPanel 的 REST fetch 在之后才执行。

**解决**：停止单点修复，启动 `/superpowers/systematic-debugging` 系统性诊断。从"数据是否在 DB → API 是否返回 → 前端 Store 是否收到 → 组件是否渲染"逐层排查，输出《缺陷-规格映射表》后才开始修复。关键认知：**当同一个功能需要修复 3 次以上，问题一定出在架构层而非实现层**。

### 问题 2：AI 的"过度实现"——在需求不清时就写复杂代码

**表现**：早期 AI 为 GuestCard 写了完整的状态机动画 + 4 种 CSS 状态 + 思考气泡 + 冷却指示器——但实际上嘉宾状态根本没有从后端推送。

**根因**：AI 被详细的 UI 设计文档"诱导"为每个设计元素都写实现代码，但后端 SSE 链路在 6 轮迭代后才打通。

**解决**：在 Prompt 中引入"分层验证路线图"——上一层（数据契约层）未通过测试，禁止进入下一层（UI 渲染层）。强制 AI 回答"这个功能的数据从哪里来？"才能开始写前端代码。关键认知：**"能不能做"比"做成什么样"更优先——没有数据源的 UI 组件只是装饰**。

### 问题 3：AI 的"遗忘曲线"——长对话中丢失早期约束

**表现**：经过 40+ 轮对话后，AI 开始在 `App.jsx` 中写硬编码的 `prompt()` 弹窗、在按钮中使用 emoji、遗忘 "禁止整页滚动" 的布局约束。

**根因**：中文语境下 Claude Code 的上下文窗口有限，早期 SDD 文档和 UI 设计契约在长对话中被挤出注意力范围。

**解决**：建立"定期回归审计"机制——每完成一个大模块，显式调用 `/superpowers/systematic-debugging` 回读 `docs/` 下的所有设计文档，生成差距表。同时在 Prompt 中显式引用文件路径（如"严格参考 `docs/ui-implementation-plan.md` Step 6"），强制 AI 重读契约。关键认知：**AI 的记忆需要"外部化"——把约束写入版控文件，每次对话显式引用，而非依赖 AI 自行记住**。

## 三、对"工程化 AI 开发"的理解

### 3.1 从"用 AI 写代码"到"用 AI 做工程"

传统认知：AI = 代码补全工具。工程化视角：AI = **开发流程的执行引擎**。

| 传统方式 | 工程化方式 |
|---|---|
| "帮我写一个数据库" | 先让 AI 输出领域建模文档 → 审核 → 再生成 SQL |
| "这里报错了，修一下" | 先让 AI 追溯调用栈 → 定位根因 → 输出测试用例 → 再修代码 |
| "界面不好看，改一下" | 先让 AI 分析设计令牌体系 → 定义色彩语义 → 分层实施 |

### 3.2 核心原则

**契约先于代码**：任何功能开发前，必须有文档化的契约（Schema / API Spec / Design Doc）。AI 基于契约生成代码，验收也基于契约。

**测试先于实现**：TDD 不是"写完代码补测试"，而是"测试定义什么是正确"。本项目 102 个测试用例（70 后端 + 32 前端）是所有重构的安全网。

**根因先于修复**：遇到 Bug 不急于改代码，先执行 `systematic-debugging` 流程——收集证据 → 形成假设 → 验证假设 → 单点修复。累计避免了至少 10 次"修了 A 坏了 B"的循环。

**审计先于迭代**：每完成一个大模块，回归 SDD 审计。本项目在中后期发现"共识引擎缺失"和"发言机械轮流"两个致命偏离，正是因为重新对照了原始需求文档。

### 3.3 AI 协同开发的角色定位

AI 不是"程序员"，而是**执行引擎 + 知识库 + 测试执行器**。

人类开发者（我）的职责：
- 定义"做什么"和"为什么"（需求分析、架构决策、优先级排序）
- 建立约束边界（SDD 文档、TDD 流程、代码规范）
- 验收与审计（读代码、跑测试、比对契约）

AI 的职责：
- 执行"怎么做"（生成代码、跑测试、格式化输出）
- 提供方案选项（"方案 A/B/C 的利弊"）
- 快速迭代（批量 TDD、全量回归、CSS 令牌替换）

**工程化 AI 开发不是让 AI 替代人做决策，而是把 AI 嵌入一个可预测、可验证、可回溯的开发流程中——人定方向，AI 跑流程，测试守底线。**
